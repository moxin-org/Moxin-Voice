## Context

当前音色管理仅提供平铺列表，缺少标准化标签，导致检索效率低且不同页面可能出现分类口径不一致。该变更涉及 UI 筛选、音色数据模型、持久化与查询逻辑，属于跨模块改动。约束条件是：需要兼容已有音色数据，并保证后续可扩展更多标签值。

## Goals / Non-Goals

**Goals:**
- 建立统一音色标签模型，覆盖性别年龄（男声/女声/童声）与风格（甜美/磁性/青年音）。
- 在音色管理界面提供基于标签的组合筛选，并实时刷新结果。
- 为现有音色数据提供向后兼容策略（默认值或空值处理），避免升级后不可用。

**Non-Goals:**
- 不在本次变更中引入新的推荐算法或排序策略。
- 不扩展到语言、口音、情绪等更多维度标签。
- 不重构非音色管理页面的全部筛选组件。

## Decisions

1. 采用双维度离散枚举标签模型
- 方案：`gender_age` 与 `style` 两个字段分别使用受控枚举值。
- 原因：前后端校验简单，一致性高，便于查询索引与国际化映射。
- 备选方案：自由文本标签。放弃原因是难以约束输入，筛选聚合成本高。

2. 筛选逻辑使用“维度内 OR、维度间 AND”
- 方案：同一维度可多选（如男声+童声），跨维度组合取交集（如男声 AND 磁性）。
- 原因：符合用户筛选心智，兼顾精确性与可用性。
- 备选方案：全局 OR。放弃原因是结果过宽，噪声高。

3. 历史数据采用可回填默认标签策略
- 方案：读取旧音色时，如果标签缺失则按配置回填默认值（或标记为未分类），并在编辑保存时写回新结构。
- 原因：避免一次性强制迁移造成发布风险。
- 备选方案：发布前一次性数据迁移。放弃原因是回滚复杂、失败影响面大。

4. 标签展示与存储解耦
- 方案：存储层使用稳定枚举 key，界面层通过字典映射显示文案。
- 原因：避免文案变更触发数据迁移，支持后续多语言。
- 备选方案：直接存中文文案。放弃原因是不可控且不利于国际化。

## Risks / Trade-offs

- [历史数据标签缺失导致筛选结果异常] -> 通过缺省回填和“未分类”兜底分组，避免数据丢失。
- [筛选组合增多带来性能开销] -> 在查询层增加标签索引或预过滤缓存，限制单次查询返回量。
- [标签枚举后续扩展频繁] -> 通过集中枚举注册点与映射配置降低改动范围。
- [前后端标签定义漂移] -> 引入共享常量或协议校验测试，确保枚举一致。

## Migration Plan

1. 扩展音色数据结构，允许 `gender_age`、`style` 字段为空。
2. 上线读取兼容逻辑：旧数据缺失标签时返回默认值/未分类。
3. 上线 UI 标签筛选与编辑能力。
4. 在音色编辑或批量维护流程中逐步回填标签。
5. 监控筛选命中率与错误日志，确认稳定后再考虑强约束非空。
6. 回滚策略：关闭筛选入口并退回旧读取逻辑，保留已写入字段不影响旧版本读取。

- 组合筛选是否需要支持“清空后显示全部”以外的快速预设？
- 是否需要在本次变更就提供批量标签编辑工具？
