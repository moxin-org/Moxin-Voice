## Context

当前语音功能可产出音频，但前端未提供“输出音色”参数控制入口，用户无法在 UI 中直接调整语速、音调。现有语音请求链路已经具备参数承载能力，因此本次优先通过前端补齐配置面板、状态管理与参数透传，避免引入后端结构性改造。

## Goals / Non-Goals

**Goals:**
- 在前端提供语速与音调可视化选项，并显示当前选中值。
- 在用户触发语音输出时，稳定将选项映射为请求参数并传入语音生成流程。
- 为未配置与越界配置提供默认值与兜底策略，保证请求可执行。

**Non-Goals:**
- 不新增新的语音引擎能力或后端合成接口。
- 不在本次变更中扩展复杂音色参数（如情感、停顿风格、发音人混音）。
- 不重构整套语音页面，只在现有设置区域做增量改造。

## Decisions

1. 使用离散选项而非自由输入滑杆
- 决策：语速、音调采用固定档位（例如 慢/中/快、低/中/高）进行选择。
- 原因：离散值更容易与现有引擎参数做稳定映射，且交互上更直观。
- 备选方案：使用连续滑杆。未采用原因是需要额外处理精度、边界和跨引擎映射一致性。

2. 选择“前端状态单一来源 + 请求时映射”
- 决策：将用户选择保存在语音设置状态中，在发起请求前统一映射为后端可识别参数。
- 原因：避免在多个组件重复转换，降低不一致风险。
- 备选方案：组件内直接输出后端参数。未采用原因是耦合高，不利于后续扩展更多 UI 表达层。

3. 默认值与回退在参数构建层统一处理
- 决策：若 UI 未产生有效选择，参数构建层回退到“中速/中调”（或项目现有默认值）。
- 原因：即使 UI 状态异常也能保证请求有效，减少运行时错误。
- 备选方案：仅在 UI 层限制。未采用原因是无法覆盖程序化触发或状态同步异常场景。

## Risks / Trade-offs

- [风险] 前端离散档位与实际引擎效果可能不完全线性对应 -> 缓解：在映射表中保留可调常量，并提供基础听感回归用例。
- [风险] 旧会话或缓存状态缺失新字段 -> 缓解：读取状态时补全默认值，避免空值进入请求。
- [权衡] 选择离散档位牺牲了高级用户的精细调节能力 -> 缓解：后续可在不破坏兼容的前提下新增“高级模式”。

## Migration Plan

1. 增加前端配置字段与默认值（语速、音调）。
2. 在语音设置 UI 增加选项控件与选中态展示。
3. 在语音请求参数构建逻辑接入映射与兜底。
4. 通过手工回归确认默认行为与各档位输出可用。
5. 如需回滚，仅移除新增 UI 和映射逻辑，保留原请求参数路径。

## Open Questions

- 当前项目对语速/音调的参数命名是否在不同语音引擎间完全一致，是否需要引擎级映射分支。
- 前端选项是否需要持久化到用户偏好，以便下次进入页面自动恢复。
