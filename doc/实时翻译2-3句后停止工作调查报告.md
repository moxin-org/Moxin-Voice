## 4. 关于 “KV cache 无限增长” 的调查结论

### 4.1 KV cache 是什么

这里的 `KV cache` 指的是自回归推理中用于加速 attention 计算的 key/value 缓存。  
在本项目的 Rust 推理实现中，这个推理概念由 Rust 类型 `KVCache` 承载。

可参考：

- [qwen3-mlx/model.rs](C:\Users\FPG_123\Documents\projects\OminiX-MLX\qwen3-mlx\src\model.rs#L743)
- [mlx-rs-core/cache.rs](C:\Users\FPG_123\Documents\projects\OminiX-MLX\mlx-rs-core\src\cache.rs#L92)

### 4.2 当前代码不支持“同一个 KV cache 跨句无限增长”的判断

在 translator 中，每次翻译都会新建一个局部 cache：

```rust
let mut cache = Vec::new();
let generator = Generate::<KVCache>::new(model, &mut cache, temperature, &prompt_tokens);
```

证据：

- [dora-qwen3-translator/main.rs](C:\Users\FPG_123\Documents\projects\Moxin-Voice\node-hub\dora-qwen3-translator\src\main.rs#L343)

这说明：

- 每次单句翻译都使用一个新的 cache 容器
- 该 cache 生命周期受当前翻译请求控制
- 现有代码没有证据表明“同一个 cache 对象在多句之间不断叠加而不释放”

因此，“2-3句后停止”目前**不应优先定性为跨句 KV cache 无限增长**。

### 4.3 但 KV cache 仍然可能导致单次推理内存压力

虽然不是“跨句无限增长”，但单次推理期间，KV cache 仍然会占用显著内存。  
尤其是长 prompt、长输出、多层模型时，单次推理内存峰值会明显提高。

因此，KV cache 仍然是内存压力来源之一，但更准确的判断应是：

- **它是单次推理期的活跃内存组成部分**
- 不足以单独证明“跨句泄漏”

## 5. 关于 `mlx_clear_cache()` 的调查结论

### 5.1 `mlx_clear_cache()` 的底层实现

MLX Rust 绑定中的 `mlx_clear_cache()` 直接调用：

```cpp
mlx::core::clear_cache();
```

证据：

- [memory.cpp](C:\Users\FPG_123\Documents\projects\OminiX-MLX\mlx-rs\mlx-sys\src\mlx-c\mlx\c\memory.cpp#L11)

同一组 memory API 还包括：

- `mlx_get_active_memory`
- `mlx_get_cache_memory`
- `mlx_get_peak_memory`
- `mlx_set_cache_limit`

证据：

- [memory.cpp](C:\Users\FPG_123\Documents\projects\OminiX-MLX\mlx-rs\mlx-sys\src\mlx-c\mlx\c\memory.cpp)
- [memory.h](C:\Users\FPG_123\Documents\projects\OminiX-MLX\mlx-rs\mlx-sys\src\mlx-c\mlx\c\memory.h)

### 5.2 对 `mlx_clear_cache()` 的语义判断

基于 API 分组与项目内用法，本次调查得出的高置信结论是：

- `mlx_clear_cache()` 清理的是 **MLX 缓存内存 / allocator cache / Metal buffer pool**
- 它不是“强制删除仍在使用的模型权重、活跃 tensor、活跃 KV cache”

### 5.3 为什么可以这么判断

最关键证据是：`qwen3-mlx` 本身就在生成循环内部低频调用 `mlx_clear_cache()`。

证据：

- [qwen3-mlx/model.rs](C:\Users\FPG_123\Documents\projects\OminiX-MLX\qwen3-mlx\src\model.rs#L836)

如果 `mlx_clear_cache()` 会删掉活跃的推理状态，那么这段库代码本身就无法正常工作。  
因此，“它会把还在使用的模型权重、活跃 tensor、活跃 KV cache 硬删掉”的说法，与当前源码证据不符。

### 5.4 如何正确理解 `active_memory` 与 `cache_memory`

建议将其理解为：

- `active_memory`: 当前仍被活跃对象引用、正在使用的内存
- `cache_memory`: 已可复用、但 MLX 暂时保留以便下次快速复用的缓存内存

因此：

- `clear_cache()` 更像“尽量把 `cache_memory` 还掉”
- 不是“清掉一切内存”

### 5.5 正确的使用方式

合理调用时机：

- 一次完整推理请求结束后调用一次
- 或在超长生成中低频调用，例如每 256 token 一次

不推荐：

- 每个 token 高频调用
- 每个小 step 调用
- 把它当成通用内存回收器频繁触发

项目内也有文档指出过于频繁会带来性能开销：

- [MLX_BEST_PRACTICES.md](C:\Users\FPG_123\Documents\projects\OminiX-MLX\qwen-image-mlx\docs\MLX_BEST_PRACTICES.md#L35)
- [OPTIMIZATION_FINDINGS.md](C:\Users\FPG_123\Documents\projects\OminiX-MLX\qwen-image-mlx\docs\OPTIMIZATION_FINDINGS.md#L153)

## 6. 对当前问题的最可能根因判断

### 6.1 当前最高优先级怀疑

当前最可能的问题不是“KV cache 对象跨句无限增长”，而是：

**`dora-qwen3-translator` 在每次翻译中产生的 MLX/Metal 缓存内存逐步积累，未及时归还，在 16GB Mac 上触发内存压力，最终导致 translator 卡死、异常退出或被系统回收。**

这与用户观察到的现象高度吻合：

- 内存先升至 `14.x GB` 黄色高压区
- 随后快速下降到 `9 GB`
- 同时实时翻译链路停止

该现象更像：

- 某个关键进程因为内存压力崩溃或退出
- 退出后缓存 / 分配被释放

而不像“纯逻辑状态错乱但进程仍健在”。

### 6.2 当前次优先级怀疑

`qwen3-asr-mlx` 也可能存在类似问题。

证据：

- ASR 内部同样会创建局部 `KVCache`，见 [qwen3-asr/model.rs](C:\Users\FPG_123\Documents\projects\OminiX-MLX\qwen3-asr-mlx\src\model.rs#L777)
- 当前未看到 ASR 在一次完整转写结束后显式执行类似 `mlx_clear_cache()` 的逻辑

因此，如果 translator 侧清缓存后问题依旧存在，应立即将 ASR 侧上升为同优先级调查对象。

### 6.3 低优先级怀疑

仍可能存在下列非内存类问题，但优先级低于内存压力：

- translator/ASR 某处 send/recv 阻塞或链路背压
- Dora 节点因下游失联导致上游同步异常
- 会话边界错配导致 translator 看似“停住”

但这些方向目前无法更好解释 “内存冲高后瞬降 + 同时停止” 这一强相关现象。

## 8. 关于 16GB Mac mini 的可行性判断

当前运行组合为：

- ASR: `Qwen3-ASR-1.7B-8bit`
- Translator: `Qwen3-8B-4bit`
- 外加 Moxin Voice App 本身的 UI / 桥接 / Dora 开销

对 16GB 统一内存 Mac mini 的判断如下：

- “能跑起来”是有可能的
- “长时间稳定持续运行”目前处在边缘状态
- 如果不做缓存治理，不看好长期稳定性
- 即使做了缓存治理，这套组合也更像“最低可用”，而非“宽松配置”

原因：

- 两个模型都需要常驻内存
- 推理过程中还有临时激活、KV cache、MLX cache memory
- macOS 统一内存会被 CPU / GPU / 系统共同争用
- 当前用户实测已进入 `14.x GB` 高压区，说明余量很小

因此，本问题的修复目标更应是：

- 尽量让缓存内存在单次请求结束后快速回落
- 避免多次请求形成“高水位平台”

而不是假设 16GB 环境本身足够宽裕。

## 9. 建议的实机调试方案

以下调试建议应在 Mac mini 上进行。

### 9.1 首先确认运行版本

确认本地实际运行的构建是否已经包含：

- `9fe804d` 中 translator 的 `mlx_clear_cache()` 逻辑

否则后续观察没有参考价值。

### 9.2 给 translator 增加 MLX 内存日志

建议在每次翻译前后记录：

- `active_memory`
- `cache_memory`
- `peak_memory`

推荐记录点：

- 进入 `translate_and_emit()` 前
- `send_translation_chunk(..., "complete", ...)` 后
- `mlx_clear_cache()` 调用前后

目标是确认：

- 是否是 `cache_memory` 持续积累
- 清缓存后是否真的回落
- 停止前是否出现明显异常峰值

### 9.3 给 ASR 增加同类日志

建议在每次 `transcribe_samples()` 前后记录同样三项：

- `active_memory`
- `cache_memory`
- `peak_memory`

若 translator 修复后问题仍在，应立即检查 ASR 是否成为新的内存高压来源。

### 9.4 同步观察 Dora 节点状态

建议在问题发生时同步观察：

- `translator` 是否先停止输出
- `asr` 是否仍在运行
- 是否有某个节点先退出或报错

重点要确认：

- 是 translator 先死，ASR 只是表面上“不再显示”
- 还是 ASR 实际先停
- 还是二者同时失联

### 9.5 验证标准

修复后建议用以下标准判断是否通过：

- 连续说话超过 2-3 句后，翻译链路不再中断
- `source_text` 和 `translation` 都可持续更新
- 不再出现高压区后立刻停机的现象
- `cache_memory` 在单次请求结束后可回落，不持续堆高

## 10. 最终结论

截至本次静态调查，可以给出如下结论：

1. 实时翻译当前最应优先怀疑的是 `dora-qwen3-translator` 的 MLX/Metal 缓存内存积累问题，而不是跨句的 KV cache 无限增长。
2. `mlx_clear_cache()` 的源码证据支持它是“清缓存内存”，不支持“强删活跃权重 / 活跃 tensor / 活跃 KV cache”这一说法。
3. translator 挂掉时，UI 会同时失去原文与译文更新，因此“ASR 也停止”的表象并不能直接说明 ASR 是首发故障点。
4. `qwen3-asr-mlx` 仍然是第二优先级可疑对象，后续必须纳入同样的内存观测范围。
5. 在 16GB Mac mini 上，这套组合不是宽松环境，必须依赖缓存治理和内存观测才能判断是否可稳定运行。

一句话总结：

**当前最可信的方向是：实时翻译在 2-3 句后停止工作，根因大概率是 MLX/Metal 缓存内存逐步积累导致的内存压力问题，translator 侧最值得优先验证，其次是 ASR 侧。**
