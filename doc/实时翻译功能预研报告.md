## 一、需求概述

为 Moxin Voice 新增实时翻译能力，核心流程如下：

```
音频输入（麦克风 / 视频文件）
  → Qwen3-ASR-1.7B（语音识别）
  → Qwen3 LLM（端侧翻译）
  → 独立浮窗实时显示（可拖拽、可调整大小、可自定义样式）
```

用户选择输入语言与目标语言，浮窗作为独立操作系统窗口叠加在其他内容之上，不影响 TTS 主功能的正常使用。

## 二、现有基础评估

### 2.1 ASR 组件：Qwen3-ASR-1.7B

**结论：可直接复用，零开发成本**

`dora-qwen3-asr` 节点已完整实现，当前已在 TTS 主流程（录音转写）中稳定使用。

| 维度 | 现状 |
|------|------|
| 实现语言 | Rust + MLX（Apple Silicon 原生） |
| Dora 接口 | 输入 `audio`（Float32Array，16kHz mono）；输出 `transcription`（StringArray） |
| 语言支持 | zh、en、ja、ko、de、fr、es、ru、ar |
| 模型路径 | `~/.OminiX/models/qwen3-asr-1.7b` |
| 推理性能 | M4 Pro 实测：RTF ≈ 5–10×（3 秒音频 ≈ 0.3–0.6s 识别时间） |
| 主要限制 | 当前为**批处理模式**（接收完整音频块），不支持流式逐词输出 |

需要在音频流入 ASR 之前插入分句层（VAD）。

### 2.2 翻译 LLM：qwen3-mlx（OminiX-MLX）

**结论：核心库已就绪，需新建 Dora 节点**

OminiX-MLX 中的 `qwen3-mlx` crate（路径 `OminiX-MLX/qwen3-mlx`）已完整实现 Qwen3 系列 LLM 推理，可直接用于构建翻译节点。

| Qwen3-8B 4-bit | `mlx-community/Qwen3-8B-4bit` | 5 GB |

#### 性能数据（M3 Max，来自 README 实测）

| 模型 | Prefill 速度 | 解码速度 | 显存占用 |
|------|------------|---------|---------|

#### API 接口（来自源码调研）

`qwen3-mlx` 提供简洁的 Rust API：

```rust
use qwen3_mlx::{load_model, Generate, KVCache};
use mlx_lm_utils::tokenizer::{
    load_model_chat_template_from_file, ApplyChatTemplateArgs, Conversation, Role, Tokenizer,
};

// 加载模型和 tokenizer
let mut tokenizer = Tokenizer::from_file(&tokenizer_file)?;
let chat_template = load_model_chat_template_from_file(&tokenizer_config_file)?.unwrap();
let mut model = load_model(model_dir)?;

// 构建翻译 prompt（chat template）
let conversations = vec![Conversation { role: Role::User, content: text }];
let args = ApplyChatTemplateArgs { conversations: vec![conversations.into()], ..Default::default() };
let encodings = tokenizer.apply_chat_template_and_encode(chat_template, args)?;
let prompt_tokens = Array::from(&encodings[..]).index(NewAxis);

// 流式生成（Iterator，每次 yield 一个 token）
let mut cache = Vec::new();
let generator = Generate::<KVCache>::new(&mut model, &mut cache, 0.0, &prompt_tokens);

for token in generator {
    let token_id = token?.item::<u32>();
    if token_id == 151643 || token_id == 151645 { break; } // Qwen3 EOS tokens
    let text = tokenizer.decode(&[token_id], true)?;
    // 实时发送到 UI
}
```

`Generate<KVCache>` 是一个 Rust Iterator，天然支持流式 token 输出，非常适合实时显示翻译进度。

#### 翻译专用 System Prompt 设计

翻译场景需抑制 LLM 的闲聊和解释倾向，使用 `temperature=0.0`（贪心解码）以确保翻译一致性：

```
System: You are a professional interpreter. Translate the following {src_lang} text into {tgt_lang}.
        Output only the translation. No explanations, no notes.
```

EOS token 检测（Qwen3）：`151643`（`<|endoftext|>`）和 `151645`（`<|im_end|>`）。

#### 模型选型建议

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 高质量翻译（主方案） | Qwen3-8B 4-bit | 翻译质量最佳，5GB 内存，M4 Pro 可流畅运行 |
| 低配设备 / 低延迟 | Qwen3-4B 4-bit | 3GB，75 tok/s，延迟更低 |
| 极低延迟场景 | Qwen3-1.7B 4-bit | 最快，但翻译质量有限 |

对于每句 30–60 tokens 的输出，Qwen3-8B 4-bit 预计翻译延迟约 **0.5–1.2 秒**。

## 三、浮窗 UI：Makepad 独立窗口方案

**结论：Makepad 原生支持多窗口，可以直接通过第二个 `Window` widget 实现独立浮窗**

### 3.1 Makepad 多窗口能力

经过对 Makepad 源码的深入调研（`widgets/src/multi_window.rs`、`widgets/src/window.rs`），确认：

- Makepad 提供 `MultiWindow` widget，专门用于管理多个独立 OS 窗口
- 每个 `Window` 实例对应一个真正的操作系统窗口（macOS NSWindow）
- 窗口具有标准 macOS 外观（标题栏、关闭/缩小/全屏按钮、可拖拽/缩放）
- 通过 `NSWindowStyleMask` 控制窗口样式（已在 `macos_window.rs` 中实现）

### 3.2 技术实现方案

**方案一（推荐）：在现有 App 中增加第二个 Window**

修改 `moxin-voice-shell/src/app.rs`，在 App live_design 中增加翻译窗口：

场景：演讲（中文）→ 实时英文字幕

| 阶段 | 说明 | 预估延迟（M4 Pro） |
|------|------|-----------------|
| 音频缓冲（VAD 等待句末静音） | 自然语速 1 句 ≈ 3–6s | 3–6s（自然等待，不影响体验） |

#### 关于 Qwen3 思考模式

Qwen3 模型默认支持"思考模式"（Chain of Thought），会在 `<think>...</think>` 标签内输出大量推理过程，不适合实时翻译场景。需要在 system prompt 中明确关闭：

```
System: /no_think You are a professional interpreter. Translate {src_lang} to {tgt_lang}. Output only the translation.
```

或通过 chat template 参数设置 `enable_thinking=false`（需确认 mlx-lm-utils 是否支持）。

## 八、句法延迟问题专项预研

### 8.1 什么是"句法延迟"

在人工同声传译中，口译员必须一边听源语言、一边用目标语言同步输出，这时会遇到一个根本性障碍：**不同语言的句法结构不同，导致关键语义成分在源语言中出现的顺序，与目标语言的表达顺序不一致**。

口译员必须在脑中"缓存"已听到的成分，等待足够的上下文后，才能开始产出目标语言。这段等待时间称为**句法延迟（Syntactic Delay）**，也叫"耳-声延迟（Ear-Voice Span, EVS）"。

关键问题：**我们的系统是否会遇到同样的问题？**

---

### 8.2 我们系统的本质：不是同声传译

在深入分析各语言对之前，必须先明确一个核心区别：

| | 人工同声传译 | 我们的系统 |
|--|------------|----------|
| 输入粒度 | 逐词实时 | **整句（VAD 分句后）** |
| 翻译时机 | 说话者说话时立即开始 | **等说话者说完一句（或停顿）后才翻译** |
| 句法延迟影响 | 非常严重 | **基本消除** |

我们的系统采用"先 VAD 分句，再整句送入 LLM 翻译"的架构。LLM 拿到的是完整句子，可以做全局句法重排，**不需要在听到一半时就开始翻译**。这从设计层面规避了传统同声传译的句法延迟问题。

然而，仍存在两类**残余风险**需要分析。

---

### 8.3 三语言对的句法结构差异分析

#### 8.3.1 中文（ZH）→ 英文（EN）：中等复杂度

**差异点 1：名词修饰语位置（最关键）**

中文是**前置修饰**（定语在名词之前），英语是**后置修饰**（关系从句在名词之后）：

```
中文：[他去年在实验室做的那个关于量子纠缠的] 实验
       ←─── 整个修饰语链 ─────────────────   名词
英文：The experiment [that he conducted in the lab last year on quantum entanglement]
      名词              ──────── 关系从句（后置）─────────────────────────────────
```

对于整句翻译，这不是问题——LLM 看到完整中文句子后，能正确输出英文句子（名词在前，从句在后）。

**对于 VAD 的风险**：如果说话者在长修饰语链中间停顿（如上例中"那个关于量子纠缠的"之后停顿），VAD 会提前切断，ASR 输出一个以"的"结尾的片段，LLM 收到的是残句：`他去年在实验室做的那个关于量子纠缠的`——没有中心名词，无法完整翻译。

**差异点 2：把字句（处置句）**

```
中文：我 把 这份报告 写完了
英文：I finished writing the report
```

处置句的宾语前置（"把"后面），动词和结果补语在宾语之后。整句翻译时 LLM 可以正确处理，无句法延迟问题。

**差异点 3：句末语气词决定句子类型**

```
中文：你去了 [吗]  → 疑问句    英文：Did you go?
中文：你去了 [吧]  → 推测疑问  英文：You went, didn't you?
中文：你去了 [啊]  → 感叹/确认 英文：So you went!
```

对于流式 token 显示：LLM 在生成英文时，通常需要先看到语气词才能确定句子类型（疑问/陈述/感叹）。如果流式输出过早，用户可能看到错误的句式（如先显示陈述句，最后被修正为疑问句）。

**严重程度（整句翻译）**：低。LLM 完全有能力处理上述结构。

---

#### 8.3.2 英文（EN）→ 中文（ZH）：较低复杂度

**差异点 1：后置关系从句 → 前置定语**

```
英文：The researcher [who won the Nobel Prize last year] is presenting today
中文：[去年获得诺贝尔奖的] 研究员 今天正在演讲
```

LLM 需要先处理整个英文句子，找到关系从句的边界，再重排为中文的前置结构。整句翻译时无问题。

**差异点 2：时态 → 体标记**

英文有丰富的时态形态（现在完成时、过去完成时等），中文用体标记（了/过/着）表达。这是语义翻译挑战，不是句法延迟问题。

**严重程度（整句翻译）**：低。

---

#### 8.3.3 中文（ZH）→ 法文（FR）：中等偏高复杂度

除了 ZH→EN 的所有问题外，法文还有独特挑战：

**特殊问题：性别歧义**

法语名词有阴阳性，形容词和冠词必须与名词性别一致。中文无性别标记：

```
中文：这位研究员非常有才华
法文（阳性）：Ce chercheur est très talentueux
法文（阴性）：Cette chercheuse est très talentueuse
```

从中文音频中，LLM 无法确定说话者指的是男性还是女性研究员（中文"研究员"性别中性）。

**处理策略**：
1. **上下文推断**：若前文已出现该人物的性别信息，LLM 可以利用（Qwen3 支持多轮上下文）
2. **保守默认**：法语中惯例使用阳性形式作为通用形式（传统语法），但现代法语趋向使用包容性写法（如 `chercheur·e`），可通过 prompt 配置
3. **用户设置**：提供"说话者性别"设置项，注入 system prompt

**特殊问题：法语否定结构**

法语否定由两部分组成（ne...pas 包住动词）：`Je ne mange pas`。这不影响整句翻译，但 LLM 须确保两个否定词都正确输出。

**严重程度（整句翻译）**：低（句法）+ 中（性别歧义）。

---

#### 8.3.4 英文（EN）↔ 法文（FR）：最低复杂度

两种语言同为 SVO 语序，后置关系从句，结构高度相似。主要差异：
- 法语形容词有前置（grand, petit, beau...）和后置（rouge, important...）两种位置
- 法语从句结构（subjonctif 虚拟式）无对应英文形态，但属语义翻译

**严重程度（整句翻译）**：极低。

---

### 8.4 严重程度汇总

| 翻译方向 | 句法结构差异 | 整句翻译影响 | 流式显示风险 | 主要风险点 |
|---------|------------|------------|------------|----------|
| ZH → EN | 中 | 低 | 中 | 语气词末位；流式输出句型不稳定 |
| ZH → FR | 中高 | 低（句法）/中（语义） | 中 | 同上 + 性别歧义 |
| EN → ZH | 低 | 低 | 低 | 时态→体标记的语义映射 |
| FR → ZH | 低中 | 低 | 低 | 同上 |
| EN → FR | 极低 | 极低 | 极低 | 虚拟式等语义细节 |
| FR → EN | 极低 | 极低 | 极低 | — |

---

### 8.5 残余风险：VAD 中途切断

虽然整句翻译架构规避了传统句法延迟，但 VAD 的分句行为引入了一个**等效风险**：

**说话者在句法未完成处停顿，VAD 提前切断。**

这是我们系统中唯一真正需要处理的句法问题。以下是高危场景：

**场景 1：中文长定语链（高危）**

```
说话者：我们需要找到 [那些在过去五年里一直坚持研究...
         ← ASR 此时输出 →
         VAD 检测到 400ms 停顿，切断！
         ...基础科学的] 年轻科学家
```

VAD 切断后，LLM 收到残句 `我们需要找到那些在过去五年里一直坚持研究`，缺少中心名词，LLM 可能输出错误翻译。

**场景 2：英文/法文从句前置（中危）**

```
说话者：Although the results were somewhat mixed, [pause] the team decided...
         ← 即使正确切断，翻译"虽然结果有些混杂"也是完整子句，可以翻译 →
```

英语/法语中，从句边界通常与自然停顿对齐（连词 although/bien que 明确标记），风险低于中文。

**场景 3：句末语气词缺失（中危，ZH→EN/FR）**

```
说话者：这个方案很有创意 [停顿，然后加]，但是成本太高了
         ↑
         VAD 可能在此切断，输出 "这个方案很有创意"
         LLM 正确翻译为 "This proposal is quite creative"
         但用户误解为这是完整评价，而非被 "but" 转折的从句
```

这不是翻译错误，但会造成语义断章取义。

---

### 8.6 应对策略

针对以上残余风险，提出分级处理策略：

#### 策略一：语言感知的 VAD 阈值（必做）

不同语言的自然停顿模式不同：

| 输入语言 | 推荐最小静音阈值 | 理由 |
|---------|---------------|------|
| 中文 | **500ms** | 中文句内停顿（换气、思考）频繁，且长定语链中没有明显停顿信号 |
| 英文 | 350ms | 英语句间停顿通常更短，结构更规律 |
| 法文 | 400ms | 介于两者之间 |

实现：在 `translation.yml` 中以 `SRC_LANG` 环境变量控制 VAD 阈值，而非使用统一阈值。

#### 策略二：句法完整性轻量校验（推荐）

在 VAD 触发后、发送给 ASR 之前，对切断文本做轻量启发式检查：

**中文检查规则**（在翻译节点 Rust 代码中实现）：

```rust
fn is_syntactically_complete(text: &str) -> bool {
    let trimmed = text.trim();
    // 以定语标记"的"结尾 → 可能是被截断的修饰语链
    if trimmed.ends_with('的') || trimmed.ends_with("的，") {
        return false;
    }
    // 以连词结尾 → 句子未完成
    let incomplete_endings = ["但是", "然而", "而且", "不过", "虽然", "尽管", "因为", "所以", "如果", "假如"];
    for ending in &incomplete_endings {
        if trimmed.ends_with(ending) {
            return false;
        }
    }
    // 以正常句末标点结尾 → 完整
    true
}
```

若判定为不完整，将该音频块与下一个 VAD 块**合并**，等待更多内容再翻译。

**英/法文检查规则**：

```rust
fn is_complete_en_fr(text: &str) -> bool {
    let t = text.trim();
    // 以从属连词开始但无主句 → 不完整
    let incomplete_starters = ["Although", "Bien que", "While", "Because", "Since", "If", "Unless"];
    // 简单启发：如果文本包含从属连词但不包含逗号（通常从句后跟主句前有逗号），可能不完整
    // 完整判断留给 LLM 的 prompt 处理
    true // 英法文风险较低，默认接受
}
```

#### 策略三：LLM Prompt 的容错指令（必做）

在翻译节点的 system prompt 中增加残句处理指令：

```
/no_think You are a professional interpreter. Translate the {src_lang} text to {tgt_lang}.
Output only the translation.
If the input appears to be an incomplete sentence fragment (e.g., ends mid-clause),
translate what is available and append "[...]" to indicate continuation is expected.
Do not fabricate content to complete the thought.
```

这样，当 VAD 不可避免地切出残句时，LLM 会输出 `"The researchers who have been studying... [...]"` 而不是虚构一个完整句子。

#### 策略四：流式显示的"暂定态"UI（推荐）

针对句末语气词造成的流式显示不稳定（ZH→EN/FR）：

**实现方案**：流式翻译期间，浮窗显示的文字使用"暂定色"（略低对比度或斜体），完成后切换为"确定色"。

```rust
// 流式中
label.apply_over(cx, live!{
    draw_text: { color: #aaaaaa }  // 暂定色
});

// 完成后
label.apply_over(cx, live!{
    draw_text: { color: #ffffff }  // 确定色
});
```

用户习惯后，自然理解正在变亮/变实的文字即为确定翻译。

#### 策略五：强制最大块时长（必做）

不论 VAD 是否检测到静音，超过 **8 秒**的音频强制切断并翻译。防止说话者长篇大论不停顿时系统无输出（用户以为系统挂了）。

即使切出残句，配合策略三的 `[...]` 标记，用户也能理解系统正在处理中。

### 8.7 结论

| 结论 | 说明 |
|------|------|
| **句法延迟问题基本不存在** | 整句翻译架构从设计上消除了传统同声传译的句法延迟问题，LLM 拿到完整句子后可做全局重排 |
| **唯一真实风险是 VAD 中途切断** | 尤其是中文长定语链，说话者在修饰语链中间停顿，导致残句送入 LLM |
| **中文输入风险最高** | 前置定语链 + 句末语气词，是中文特有的两类高危结构 |
| **EN↔FR 几乎无风险** | 两种语言结构高度相似，整句翻译效果极好 |
| **四项缓解措施可全部代码化** | 语言感知 VAD 阈值、句法完整性校验、LLM prompt 容错指令、强制最大块时长，均可在 dora-qwen3-translator 节点中实现，不影响主架构 |
| **流式显示"暂定态"** | 可降低句末语气词引起的翻译"闪回"对用户的困惑 |

**对现有方案的修改影响**：无需改变整体架构，上述策略均可在翻译节点（`dora-qwen3-translator`）内部和 `translation.yml` 配置层实现。

## 九、结论

### 可行性评估

| 维度 | 评估 |
|------|------|
| 技术可行性 | ✅ 完全可行。ASR 零成本复用，LLM 推理库已就绪，Makepad 多窗口原生支持 |
| 延迟 | ✅ 感知延迟 1–2 秒，适合实时字幕场景 |
| 内存 | ✅ M4 Pro 64GB 无压力；16GB 设备使用 4B 模型 |
| 句法延迟 | ✅ 整句翻译架构从设计上消除；VAD 残句风险可通过语言感知阈值 + 句法校验 + prompt 容错处理 |
| 开发复杂度 | 中等。主要工作在翻译节点和浮窗 Widget |

### 关键技术决策

1. **翻译后端**：本地端侧，使用 OminiX-MLX 的 `qwen3-mlx` crate + Qwen3-8B 4-bit 量化模型
2. **浮窗形式**：Makepad 独立 `Window`（OS 级窗口），不在主窗口内实现
3. **流式显示**：利用 `Generate<KVCache>` Iterator 实现 token 级实时更新；流式中使用"暂定色"，完成后切换为"确定色"
4. **VAD 分句**：语言感知阈值（ZH: 500ms、EN: 350ms、FR: 400ms）+ 8s 强制切断；结合句法完整性轻量校验防止残句
5. **思考模式**：System prompt 加 `/no_think` 前缀关闭 Qwen3 的 CoT 输出
6. **句法延迟**：整句翻译架构从设计上消除传统同声传译的句法延迟；VAD 残句通过 prompt 容错指令（`[...]` 标记）处理
